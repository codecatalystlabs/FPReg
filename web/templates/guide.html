HMIS MCH 007: Integrated Family Planning Register

This digital system implements the Uganda Ministry of Health HMIS Form 007 – Integrated Family Planning Register (Print Version July 2024). This guide explains how to use the system and how each field maps to the paper register.

1. Purpose of the Register

The Integrated Family Planning Register records all family planning client visits at a health facility. It captures:

  • Client identification and demographics
  • Visit type (new user or revisit)
  • HIV testing status
  • FP counseling delivered
  • Contraceptive methods dispensed (pills, condoms, injectables, implants, IUDs, permanent methods, FAM)
  • Post-pregnancy FP timing
  • LARC removals
  • Side effects reported
  • Cancer screening and STI screening
  • Referrals

Every client accessing family planning services must be registered, including counseling-only visits and visitors who will not return for refills.

Cancer screening captures cervical cancer (method, status, treatment if positive) and breast cancer screening.

Record STI screening status and any referrals made.

4. Skip Logic Rules

RuleBehaviour
New User / RevisitMutually exclusive. Only one can be selected. Selecting one unchecks the other.
Previous MethodOnly shown when "Revisit" is ticked.
Switching ReasonOnly shown when "Switching Method" is ticked.
Sex = MaleHides tubal ligation, cervical cancer screening, breast cancer screening. Shows vasectomy.
Sex = FemaleHides vasectomy. Shows tubal ligation, cervical and breast cancer screening.
Cervical Cancer TreatmentOnly shown when cervical cancer status is "Positive" (code 2).
Implant Removal TimingOnly shown when an implant removal reason is selected.
IUD Removal TimingOnly shown when an IUD removal reason is selected.
Client NumberNot generated for visitors. If "Visitor" is ticked, no client number is assigned.

Skip logic is enforced both on the form (frontend) and in backend validation to prevent invalid data.

5. Client Number Generation

Client numbers are automatically generated when a new registration is saved (unless the client is a visitor).

Format: {FacilityPrefix}{YYMMDD}{NNN}

Example: DHC260314001 = Demo Health Centre, 14 March 2026, 1st client of the day.

  • Facility Prefix: 2–5 character code configured for each facility (e.g., DHC, MRH).
  • Date: 6-digit date in YYMMDD format.
  • Sequence: 3-digit daily sequence number starting at 001, reset per facility per day.

The system uses database-level row locking to prevent duplicate numbers even under concurrent requests.

Full interactive API documentation is available at {{.BasePath}}/swagger/index.html.

Offline Sync Recommendation

For mobile apps that need offline support, we recommend:

  • Cache option sets locally on app start (they rarely change)
  • Queue registrations locally when offline
  • Sync queued records when connectivity is restored using POST /api/v1/registrations
  • Use the client number response to update local records with the server-assigned number
